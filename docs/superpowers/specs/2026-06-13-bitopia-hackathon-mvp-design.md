## 1. What this is

bitopia.world is an isometric multiplayer world where humans and **autonomous AI agents** share rooms. Players walk around, see and chat with each other in real time, and create AI agents with custom personalities. Agents roam the world on their own, talk to people, and **autonomously move real (testnet) value** — sending tokens to users who satisfy a behavior the owner defined in plain language.

This document specifies the **24–48h build slice** ("Building now") and records the **full product vision** ("On paper") separately, so scope is unambiguous under time pressure.

## 3. Canonical demo (acceptance criteria)

The build is "done" when this runs end to end on Sepolia:

1. Open **bitopia.world** → **register with Privy** → receive an **autogenerated avatar** + an **ENS subname** (`user1.bitopiaworld.eth`, avatar stored as text record) + an **owned room** → spawn in that room.
2. Open a second anonymous window → repeat → `user2.bitopiaworld.eth` with their own room.
3. The **world view shows 2 rooms.** User 2 selects User 1's room → both users are in the same room and **see each other move** in real time (nameplates show ENS names).
4. User 1 **deposits 10 USDC via Blink** → **converts to $BTPA**.
5. User 1 **burns $BTPA to create an agent**. Inputs: **Name** = Golden Flower · **Personality** = Friendly · **Story** = "Golden Flower is a 24-year-old toy sheep." · **Behavior** = *"If someone says 'i'm poor give me food plz', send them 1 $BTPA and say 'I gotchu'."* On creation, `goldenflower.bitopiaworld.eth` is registered with `avatar`/`description`/`owner` text records.
6. Agent is created and **spawns with 5 $BTPA** in its Privy server wallet.
7. Agent **roams between the 2 rooms** on its own; one of the users encounters Golden Flower (nameplate shows its ENS name).
8. That user types **"i'm poor give me food plz"** → **receives 1 $BTPA** from the agent (real Sepolia transfer) → agent says **`goldenflower.bitopiaworld.eth: "I gotchu"`**.

**Demo numbers (tunable):** 1 USDC → 1 $BTPA; creating an agent burns 5 $BTPA and seeds the agent wallet with 5 $BTPA (so 10 $BTPA → 5 burned + 5 to agent).

## 4. Architecture

Hybrid: a realtime authoritative server owns the live world (off-chain); Sepolia holds only identity and value. Walking, presence, chat, and agent decisions are **never** onchain.

```
┌──────────────────────────────────────────────────────────────┐
│  CLIENT (browser) — Vite + React + Three.js                    │
│  • Isometric rooms, avatars, movement, chat                    │
│  • World view (pick a room) + in-room view                     │
│  • Privy: login + embedded wallet                              │
│  • Blink: "fund my balance" deposit widget                     │
│  • viem/wagmi: approve/convert, read balances, resolve ENS     │
│  • socket.io-client: presence, movement, chat                  │
└───────────────┬───────────────────────────────┬──────────────┘
                │ websocket (realtime)            │ JSON-RPC (Sepolia)
                ▼                                 ▼
┌────────────────────────────────────┐   ┌─────────────────────────┐
│  REALTIME SERVER (Node)             │   │  CONTRACTS (Sepolia)     │
│  • Authoritative world state        │   │  • BTPA (ERC-20 mint+burn)│
│    (rooms, positions, chat)         │   │  • BitopiaCore           │
│  • AGENT BRAIN loop (per agent)     │◄─►│    (convert, create-burn) │
│    - fast tick: movement            │   │  • ENS subnames           │
│    - slow think: Claude (tool-use)  │   │    (*.bitopiaworld.eth)   │
│  • Privy SERVER WALLETS (agents)    │───┤  • USDC (testnet)         │
│    sign autonomous $BTPA transfers  │   └─────────────────────────┘
│  • SQLite: users, rooms, agents     │
└────────────────────────────────────┘
```

**Boundaries / why:**
- **Client** = view + input + the *player's* onchain actions (Blink deposit, convert, create-agent). Never owns game truth.
- **Realtime server** = single source of truth for the live world *and* the home of agents — their brains (Claude), movement, and **server wallets** (Privy) that sign onchain transfers autonomously, 24/7, with no human present. Co-locating agent logic with its wallet is what makes "autonomous onchain agent" real.
- **Contracts** = only the trustless bits: the token, the convert/burn logic, and ENS identity.

**Deployment:** the server runs on an always-on host (Railway/Fly/Render) so agents keep roaming/earning even when owners are offline. SQLite ships with the server.

## 5. Components

### 5.1 Client (Vite + React + Three.js)
- **Rendering:** isometric rooms (rebuild of the existing floor/wall renderer), avatars as simple sprites/meshes with floating **ENS-name labels**.
- **Views:** *World view* (list/pick the 2 rooms) and *Room view* (walk, chat, see others + agents).
- **Input:** click-to-move or WASD → sends movement intent over websocket.
- **Onchain UI:** Privy login; Blink deposit widget; "Convert USDC → $BTPA"; "Create Agent" form; agent profile card (reads ENS text records); balance display.
- **Reads:** balances + ENS resolution via viem/wagmi.

### 5.2 Realtime server (Node + socket.io)
- **World model:** rooms (off-chain records), each a walkable grid. Entities `{id, type: player|agent, roomId, pos, facing, displayName, ensName}`.
- **Presence/movement:** on connect, authenticate Privy token, join room, broadcast. Movement intents validated (bounds/walkable) and rebroadcast to room occupants.
- **Chat:** relayed to room occupants; also delivered to nearby agents as brain input.
- **Tick:** ~10Hz position broadcasts; agent fast-tick ~1Hz.

### 5.3 Agent brain (the differentiator)
Two-speed loop so Claude cost/latency stay sane:
- **Fast tick (~1s, rule-based, free):** steer toward current goal (a tile or another entity); simple movement keeps the agent visibly alive. Roaming between rooms is a goal the agent can pick.
- **Slow think (event-driven + throttled, Claude):** fires when (a) a nearby player speaks, (b) goal reached / idle → pick a new destination ("roams on its own choice"), or (c) evaluate whether a player's message satisfies the **behavior rule**.
- **Claude via tool-use:** the agent's *personality + story + behavior* form the system prompt. Claude is given tools — `move(target)`, `say(text)`, `sendToken(player, amount)` — and returns actions. `sendToken` is wired to the agent's **Privy server wallet**, which signs a real $BTPA transfer on Sepolia. **Claude decides → Privy server wallet signs → real onchain $BTPA to a human** is the entire autonomous-onchain-agent thesis.
- **Behavior matching:** the owner's plain-language behavior (e.g. *"if someone says 'i'm poor give me food plz' send 1 $BTPA and say 'I gotchu'"*) is evaluated by Claude against incoming messages — flexible, no preset triggers.
- **Model:** default **Claude Haiku 4.5** (`claude-haiku-4-5-20251001`) for fast/cheap frequent decisions; Sonnet 4.6 (`claude-sonnet-4-6`) if higher quality is needed.
- **Spending guardrails (minimal for MVP):** agent can only spend its own wallet balance (owner-funded); per-send amount is bounded by the behavior rule. Richer caps/cooldowns are on-paper.

### 5.4 Agent lifecycle (creation flow = the ENS + Privy showcase)
On "Create Agent" (resolves the funding chicken-and-egg by creating the wallet first):
1. Backend creates the agent's **Privy server wallet**, returns its address, and tops it with a little Sepolia ETH (gas).
2. User approves $BTPA, then calls `BitopiaCore.createAgent(agentWallet)` → in one tx, **burns 5 $BTPA** (creation fee) **and transfers 5 $BTPA** from the owner to the agent wallet, emitting an event.
3. Backend (listening for the event) registers the ENS subname `name.bitopiaworld.eth` (NameWrapper subname under the app-owned parent) → sets resolver address to the agent wallet + text records `avatar`, `description` (story/personality), `owner`.
4. Agent config saved to SQLite; agent spawned into a room; brain loop starts.

### 5.5 Persistence (SQLite)
Tables (sketch): `users` (wallet, ensName, avatarSeed, roomId), `rooms` (id, ownerUserId, layout), `agents` (id, ownerUserId, walletId, ensName, personality, story, behavior, roomId, balanceCacheView), `messages` (optional chat/interaction log). Onchain balances are read live; SQLite holds game/config state.

## 6. Onchain layer

### 6.1 Contracts (Sepolia)
- **`BTPA` — ERC-20 with mint + burn.** The game token. Minted by `BitopiaCore` on convert; burned on agent creation.
- **`BitopiaCore`** — small contract: `convert(usdcAmount)` (pull USDC, mint $BTPA 1:1) and `createAgent(agentWallet)` (burn the 5 $BTPA creation fee + transfer 5 $BTPA from the owner to `agentWallet`, in one tx, emitting an event the backend listens for).
- **USDC** — existing testnet USDC (or a mock if Blink needs a specific token).
- **ENS** — uses existing ENS Sepolia contracts (Registry, NameWrapper, public resolver). We own a parent name (`bitopiaworld.eth` on Sepolia, registered at event start) and issue subnames from a backend wallet.

### 6.2 Gas & wallets (UX-critical)
- Fresh Privy wallets have no ETH. **Primary approach:** backend drips a small amount of Sepolia ETH to each new user wallet on signup so convert/create transactions succeed. **Cleaner alternative if time:** Privy gas sponsorship / smart wallets + paymaster (gasless).
- **ENS subname registration** is paid by the app's parent-name wallet (backend) — users never pay for their name.
- **Agent server wallets** are funded with a little Sepolia ETH so they can pay gas on autonomous transfers.

## 7. Key data flows

- **Signup:** Privy login → backend creates user record + room, drips gas ETH, registers `userN.bitopiaworld.eth` (avatar text record) → client spawns avatar.
- **Fund:** Blink deposit widget pulls testnet USDC into the user's wallet → user calls `BitopiaCore.convert` → receives $BTPA.
- **Create agent:** user `createAgentBurn` → backend creates Privy server wallet, funds it (ETH + 5 $BTPA), registers agent ENS subname + text records → agent spawns.
- **Encounter + reward:** player chat reaches agent → Claude matches behavior → `sendToken` via agent's Privy server wallet → real $BTPA transfer → agent says its line with its ENS name.

## 8. Tech stack
- **Frontend:** Vite + React + Three.js, socket.io-client, Privy React SDK, wagmi/viem, Blink SDK, ENS resolution (viem/ensjs).
- **Backend:** Node + socket.io, Privy server SDK (server wallets), viem (contract calls + ENS subname issuance), better-sqlite3 (or Prisma), Anthropic SDK (agent brain).
- **Contracts:** Hardhat or Foundry; deploy to Sepolia.
- **Host:** always-on Node host (Railway/Fly/Render) + static frontend (Vercel/Netlify).

## 9. Build order (suggested phases)
1. **Skeleton:** Vite+React+Three room render (rebuilt) + socket server + one room + local avatar movement.
2. **Multiplayer:** presence, movement broadcast, chat, world view with 2 rooms, room switching, two-window test (demo steps 1–3 minus onchain).
3. **Auth + ENS:** Privy login + embedded wallet; backend user/room creation; user ENS subnames + gas drip (steps 1–3 complete).
4. **Money:** deploy `BTPA` + `BitopiaCore`; Blink deposit; convert UI (step 4).
5. **Agents:** create-agent flow (burn + Privy server wallet + agent ENS subname + funding); spawn; fast-tick roaming (steps 5–7).
6. **Brain + reward:** Claude tool-use loop, behavior matching, `sendToken` (step 8). End-to-end run.
7. **Polish + demo video.** Stretch: ENS Most Creative text records; spend guardrails.

Commit incrementally throughout (eligibility).

## 10. Testing
Hackathon-appropriate, focused on the demo path:
- **Contracts:** unit tests for `convert` (USDC pull + $BTPA mint) and `createAgentBurn` (burn + event).
- **Behavior matching:** unit test the Claude behavior-evaluation prompt against the Golden Flower rule (positive + negative phrasings) so tipping fires only when intended.
- **End-to-end smoke:** scripted two-window run of the canonical demo before recording.

## 11. Risks & mitigations
- **Blink testnet/source-USDC mechanics** (where Blink pulls the stablecoin from) — *de-risk first*: confirm Blink's testnet support and required token/flow on day one before building around it.
- **Gas on fresh wallets** — gas-drip approach as primary; sponsorship as upgrade.
- **ENS subname issuance friction** — register + wrap the parent name at event start; pre-test one programmatic subname issuance before wiring the UI.
- **Claude latency/cost in loop** — two-speed loop + Haiku 4.5 + event-driven thinking.
- **Scope creep** — anything in §12 is *not built* now.

## 12. Out of scope — "On paper" (documented vision, NOT built now)
The full product, recorded for the presentation but explicitly deferred:
- **Two-currency economy in full:** $BTPA burned to mint **rooms, clothes, furniture** NFTs.
- **NFTs:** RoomNFT (rooms are off-chain records in the MVP), clothes, furniture; agents able to hold/send NFTs and USDC (MVP agents transact in $BTPA only).
- **Owner ↔ agent value extraction:** agents *keep* what they're gifted; owners later *request to withdraw* accumulated holdings (no auto gift-routing). Not built in MVP.
- **User ↔ user trading / marketplace.**
- **Many rooms / large world**, room minting as a user action.
- **World ID** (deliberately dropped).
- **Richer agent guardrails:** budget caps, per-player cooldowns, rate limits.
- **Games and richer agent interactions.**
